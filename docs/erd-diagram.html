1. Core Property and Transfer ERD

Core Domain

users -> properties

  • One user can own many properties.
  • One property belongs to one primary owner at a time.

properties -> transfer_requests

  • One property can have many transfer requests over time.
  • Each transfer request belongs to one property.

properties -> ownership_history

  • Ownership history stores previous and new owner transitions for a property.
  • Each transfer completion updates ownership history.

2. Voting, Succession, and Co-Ownership ERD

Workflow Data

Registration voting

  • One property has one registration case in reg_blockchain_cases.
  • One registration case has many LRO votes in reg_blockchain_votes.

Succession flow

  • One succession request belongs to one property.
  • One succession request has many heirs.
  • Heirs are linked back to family_members.

Co-ownership

  • Approved succession can produce many property_co_owners for one property.
  • Consent requests and votes are stored separately for joint-owner actions.
  • M:N exists between users and properties through property_co_owners.

3. P2P and Operations ERD

Support Data

P2P communication

  • channel_participants tracks users inside negotiation channels.
  • channel_messages stores the actual chat and attachments.
  • M:N exists between users and channels through channel_participants.

Audit and monitoring

  • audit_logs stores action history by user and target.
  • system_alerts, system_health_snapshots, and system_job_runs support monitoring.

Backup and restore

  • system_backups stores backup metadata.
  • system_restore_runs links restore operations back to the backup source.
Exact Diagram Type: This is a Logical Entity Relationship Diagram (Logical ERD) because it shows database entities, primary keys, foreign-key relationships, and major business data groupings of the system. It is logical rather than physical because it models data structure, not deployment hardware.

Cardinality Summary: Your ERD includes 1:1, 1:M, and M:N relationships. Many-to-many relationships are implemented through junction entities such as property_co_owners, property_co_owner_consent_votes, and channel_participants.

Note: notifications in your project are generated dynamically from workflow state and are not stored in a dedicated notifications table, so they are not shown as a standalone entity here.
